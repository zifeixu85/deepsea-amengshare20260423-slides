01
Principle · 01

先让用户看懂，
再让用户觉得高级

  • 落地页的第一任务是信息传达，不是审美表达。
  • 用户首先想知道：你是做什么的、对我有什么用、我下一步该做什么。
  • 文案表达 优先于 视觉表达。
  • 清楚比好看更重要。
钉子句 · Quotable

"很多 AI 做出来的页面，不是丑，
而是没说清楚。"

"设计不是给产品化妆，
是帮产品把话说明白。"

Principle 01 of 06 说人话 > 看起来对
02
Principle · 02

首屏只做一件事：
用一句话讲清产品价值

  • 首屏要快速回答三个问题：这是什么 / 适合谁 / 为什么值得试。
  • 常见问题：说了很多但没有重点、讲功能不讲结果、讲自己做了什么不讲用户得到什么。
  • 一个清晰首屏的基础结构：主标题 + 副标题 + CTA + 辅助证明。
基础结构 · Skeleton

H1  一句话讲清你是什么
P   给谁 / 解决什么 / 怎么不一样
CTA 现在能做的下一步
Proof 客户 logo / 数据 / 截图

Principle 02 of 06 5 秒内回答 3 个问题
03
Principle · 03

页面不是信息堆砌，
而是引导用户做决定

  • 落地页是说服顺序，不是把内容放满。
  • 用户走过的心理路径：看懂 → 产生相关性 → 建立信任 → 愿意行动。
  • 页面结构不只是排版问题，更是信息顺序问题。
说服阶梯 · Funnel

① 看懂 — Hero 区
② 相关 — 痛点 / 场景
③ 信任 — 案例 / 数据 / 背书
④ 行动 — CTA / 价格 / 风险消除

Principle 03 of 06 顺序就是设计
04
Principle · 04

用户不是没耐心，
而是没有义务理解你

  • 用户不会替你脑补，也不会主动帮你理解产品。
  • 能提前表达清楚的，不要留给用户自己猜。
  • 可落地的细节：标题尽量明确 · CTA 尽量具体 · 少用黑话 · 一屏不要太多同等级重点 · 模块标题让人一眼知道在讲什么。
反例 → 正例

了解更多 → 查看完整定价

极致体验 → 一次构建，30 秒部署到 12 个区域

Principle 04 of 06 不要让用户去翻译你
05
Principle · 05

视觉的作用是帮信息分层，
而不是制造热闹

  • 很多 AI 页面问题不是内容少，是视觉很热闹但重点不清楚。
  • 好的视觉不是元素更多，是重点更明确。
  • 视觉至少要服务于：告诉用户先看哪里 / 区分主次 / 把相关内容组织在一起 / 让阅读更轻松。
  • 一个区域里，最好只有一个视觉主角。
检查清单 · Checklist

□ 用户的眼睛 2 秒内能落到主行动？
□ 一屏里是不是有 ≥3 个同样高亮的元素？
□ 移除所有装饰，骨架还成立吗？
□ 颜色是在分层 还是在抢戏？

Principle 05 of 06 一个区域，一个主角
06
Principle · 06

Landing 不是艺术作品，
它首先是一个转化工具

  • 最终标准不是自己觉得好看，而是用户是否更容易理解、信任、行动。
  • 设计不是装饰层，是业务层的一部分。
  • 如果一个页面很好看，但没人点、没人注册、没人留下来——那它还没有真正设计好。
最后一问 · Final Question

"上线两周后回头看，
这个页面有没有真的
把人接住？"

Skill 01 · Anthropic 官方

/frontend-design

github.com/anthropics/skills ↗

"先选一个 bold 的美学方向，再开始写代码。"
这是一份给 AI 的设计自我意识。

  • 核心命令：直接挂载，作为系统级提示。任何写前端的请求，都先经过它。
  • 能达到的效果：让 AI 在动手前先选定方向（editorial / brutalism / bento / luxury / scrollytelling 等），而不是无脑出 Inter + 紫渐变 + card grid。
  • 关键 BAN list：Inter / Roboto / Arial、紫色渐变、purple-on-white、card-on-card、generic SaaS。
  • 地位：它是"为什么"。后面的 impeccable / taste-skill 都建立在这个共识之上。
# 安装方式（最简）
$ cd ~/.claude/skills
$ git clone <repo> frontend-design

# 触发：写任意前端时自动加载
› "做一个 hero section"
→ skill loaded ✓
→ 选定方向：editorial / dark
→ 选定字体：PP Editorial New + Inter
→ 拒绝：Inter as display, purple gradient
用一句话总结

"教 AI 先有审美，后写代码。"

/frontend-design · 核心内容拆解

它到底讲了什么？

主张只有一句：写代码之前，先做出一个明确的美学决定。 展开之后，分三步 + 一条禁令。

第一步 · 先想清楚

想清楚
这四件事

  • 给谁 · 用户、情境、当下心智
  • 要什么感觉 · 界面要"长成什么样"
  • 有什么约束 · 框架 / 性能 / 可访问性
  • 差异点 · 让人记住的那件事

这四件事没答清楚，不要动键盘。

第二步 · 选一个方向

挑一个方向
别骑墙

极简克制 极繁混沌 复古未来 有机自然 奢华精致 玩具感 杂志编辑 粗野主义 装饰艺术 柔和粉彩 工业功能

11 种方向任选其一，做到极致。
不做"安全的中间态"。

第三步 · 逐条打磨

5 个维度
每条都要到位

  • 字体 · 用有个性的字，标题配正文要选一对
  • 颜色 · 一个主色 + 锐利点缀 > 均匀调色板
  • 动效 · 一个高光时刻 > 满场碎片细节
  • 空间 · 不对称 / 重叠 / 破格 / 大留白
  • 细节 · 噪点 / 渐变 / 几何 / 戏剧阴影

维度不全，就不算"打磨完"。

绝不要做
Inter / Roboto / Arial / 系统默认字 紫色渐变配白底 千篇一律的卡片网格 没有方向的"通用设计"
Skill 02 · Paul Bakaus · 主讲

/impeccable

impeccable.style · github.com/pbakaus/impeccable ↗

把 frontend-design 的"思想"，
变成一条可以一步步走的流水线。

  • 本质：1 个扩展 skill + 18 个命令 + 反模式 CLI（npx impeccable detect）。
  • foundation：/impeccable 自身是底座，所有其他命令都共享它的设计原则与反模式库。
  • 三个起手：/impeccable teach（项目首次配置）· /impeccable craft（端到端实现）· /impeccable extract（沉淀设计系统）。
  • 最大价值：18 个命令每个都对应一种"AI 味"的具体修法，拿到哪条用哪条。
# 一次完整工作流（理想路径）
› /impeccable teach // 项目配置一次
› /shape     // 想清楚再开始
› /impeccable craft // 端到端实现
› /critique  // 自我评分
› /audit     // 技术体检 P0-P3
› /polish    // 上线前最后一遍

# 出现问题时
› /bolder | /quieter  // 调强弱
› /distill  // 删到只剩骨头
› /delight  // 最后加一点人味
用一句话总结

"把"做一个好页面"
拆成 18 个可复用的动作。"

零基础先掌握这 4 个

18 个命令学不完？
先会这 4 个就够用

刚开始用 AI 写前端，不需要 18 个全记。下面这 4 个打通 80% 的日常场景。

01 /impeccable teach 项目初始化
什么时候用
新项目刚开始。整个项目只需要做一次。
怎么用
直接输入 /impeccable teach。它会问你几个问题：做给谁、什么品牌调性、偏好方向。
会得到
一份 .impeccable.md 写在项目根目录。之后所有命令都会自动读它，AI 不会再乱飞。
02 /impeccable craft 主力造页面
什么时候用
要做一个新页面、新组件、新模块。
怎么用
直接讲需求，例：/impeccable craft 一个 SaaS 定价页 或 /impeccable craft 首页的 Hero 区。
会得到
它会先想再写：问清用户→选方向→出代码→打开浏览器看→不满意自己改，直到看起来像产品。
03 /critique 挑毛病
什么时候用
页面做完了，但心里觉得"差点意思"、"有 AI 味"但说不上哪里。
怎么用
/critique 当前首页。它会分两路评审：一路按 Nielsen 十大原则，一路跑反模式扫描。
会得到
一份清单：哪里像 AI、为什么像、该用下面哪个命令改。
04 /polish 最后一遍
什么时候用
功能都对了、页面也不丑了，上线前那最后一遍。
怎么用
/polish 定价页 或 /polish 整个站点。
会得到
对齐、状态（hover / focus / error）、文案一致、动效平滑、主题色规范 —— 6 个维度逐条收尾。
造页面阶段 · 把页面先搭出来

当 /impeccable craft 不够用时，
你需要这些"单点工具"

craft 是通用大招，下面这几个是专项命令——只动一件事、改得更可控。

/shape 写代码前
什么时候用
接到一个模糊需求（"做个定价页"），还不清楚要几个档位、放哪些模块。
怎么用
/shape 定价页 要 3 档 面向中小团队 — 它会先给出信息架构和 wireframe 思路，不会直接写代码。
/layout 空间节奏
什么时候用
页面看起来"挤"或"空"，各区块间距像在打架。
怎么用
/layout hero 区 — 它会调整间距、对齐、分组，让留白有节奏感。
/typeset 字体层级
什么时候用
看不清层级——标题/副标题/正文一个样，读着累。
怎么用
/typeset 整页 — 建立 display/h1/h2/body 的字号对比与行距，强化阅读路径。
/colorize 加策略色
什么时候用
灰白为主，太平、没记忆点，想加一个强调色但不知放哪。
怎么用
/colorize 整页 以琥珀色为主色 — 它会按 60-30-10 分布稀用，不是全涂一遍。
/clarify UX 文案
什么时候用
界面需要用户"猜"意思，或者文案在喊口号但没说清楚做什么。
怎么用
/clarify hero 文案 — 用动词开头，把"能得到什么"写清楚。
/impeccable extract 沉淀 token
什么时候用
写了几个页面后发现颜色、间距在重复，但没落到设计 token。
怎么用
/impeccable extract src/ — 扫代码、沉淀成 CSS 变量并回写 .impeccable.md。
调校 + 上线阶段 · 推到"真的像产品"

页面已经搭出来了，
接下来是"调强度 + 收细节"

这批命令不负责"造"，只负责"调"和"检"。能让同一版设计多走一里路。

/distill 删到骨头
什么时候用
信息堆在一起没主角；视觉/文案都想删但不知道删哪个。
怎么用
/distill hero 区 — 删装饰元素、合并重复信息，保留最关键那一条。
/animate 有目的动效
什么时候用
页面没"活"气；或者动效太多、太装饰。
怎么用
/animate hero 入场 — 只对关键信息加淡入和位移，不做无意义装饰。
/harden 真实数据
什么时候用
本地漂亮，一接真实数据就崩——超长字符串、空列表、错误状态全没考虑。
怎么用
/harden 订单列表 — 补空状态 / loading / error / 极端数据的样式。
/optimize 性能
什么时候用
Lighthouse 分数低；首屏慢、交互卡。
怎么用
/optimize 首页 — 给出 LCP / INP / 包大小的可改项和优先级。
/adapt 多端适配
什么时候用
桌面好看，手机塌；或要把同一设计适配到邮件 / 嵌入 / 小屏。
怎么用
/adapt 移动端 — 不是缩桌面，是为窄屏重新布局并确保触达。
/normalize 拉回体系
什么时候用
页面跟设计系统走偏了——颜色 / 间距 / 圆角都在"凭感觉"。
怎么用
/normalize 整页 — 把 hard-coded 值替换成 token，让页面回到体系里。
深讲 · 上线前的"质量三件套"

/critique
/audit
/polish

三个动词的分工很清楚：
评审 → 体检 → 抛光。先后顺序不能错。

  • /critique · 是不是有 AI 味：跑两路并行评审 — LLM 走 Nielsen 10 启发式，detector 跑 25 条反模式扫描。出"哪里像 AI slop"。
  • /audit · 技术上撑不撑得住：5 个维度（A11Y / 性能 / 主题 / 响应式 / 反模式）打 0-4 分，每条结论给 P0-P3 严重度。出一份可塞进 Linear 的清单。
  • /polish · 最后一遍抛光：6 个维度（对齐 / 字体 / 颜色 / 状态 / 动效 / 文案）。前提是功能已经完整 — 不然抛光就是浪费工时。

写出来之后，
怎么"调味"

  • /distill：删。竞品按钮太多、信息太满、字体三种用不下去——它问一个问题：这个页面到底要做的"那一件事"是什么？剩下不为这件事服务的，全部砍。
  • /delight：在功能完整之后，加那一点"人味"。空状态的一句话、loading 的微文案、成功反馈那一下子。规则是：把这点 delight 删掉，功能仍然 100% 可用。
  • /animate：只动 transform 和 opacity，缓动只用 ease-out exponential。如果你在 animate width/top/left，它就在做错事。
  • /clarify：改 UX 文案。"了解更多" → "查看完整定价"。错误信息不甩锅给用户，按钮描述结果不描述动作。
/distill 的灵魂之问

"这个页面，
那一件事是什么？"

剩下的全部要为它让路。

/delight 的硬规则

"删掉 delight，
功能仍然 100% 可用。"

没有任何东西"依赖"那一笑。

/clarify 的写作原则

"按钮描述结果，
不描述动作。"

「保存修改」> 「OK」。

不只是"设计审美"——

18 类规则 · 一次 audit 全查

不只审美 ·
代码 · 无障碍 · 性能 · 表单 · 国际化 · 全都管

下面是最常踩的几类。真正被扫描的共 18 类、几百条规则。

A11Y

可访问性

给所有人能用

  • 图标按钮必须有 aria-label
  • 输入控件必须 <label> 或 aria-label
  • 图片必须有 alt 和尺寸
  • 交互元素必须可键盘操作
  • 异步更新要 aria-live="polite"
PERF

性能

不卡、不爆首屏

  • 列表 > 50 项必须虚拟化
  • 渲染中严禁 getBoundingClientRect 等布局读
  • CDN 资源域加 <link rel="preconnect">
  • 图片显式 width/height 防 CLS
MOTION

动画

不晕、不过载

  • 遵守 prefers-reduced-motion
  • 只动 transform 和 opacity
  • 永不用 transition: all
  • 动画必须可中断
FORMS

表单

填得顺

  • 必配 autocomplete 和语义 name
  • 用正确的 type / inputmode
  • 永远不要阻止粘贴（密码/邮箱别拦）
  • 错误内联显示，提交时聚焦第一错项
SSR

水合安全

服务端渲染不闪

  • SSR 首帧不得依赖 window / document
  • 服务端与客户端输出必须一致
  • hydration mismatch 零容忍
